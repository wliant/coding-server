# Research: Enhanced Task Submission

**Feature**: 006-enhance-task-submission
**Date**: 2026-03-06
**Status**: Complete

## Decision 1: DB-Backed Agent Registry Table

**Decision**: New `agents` table with UUID PK, `identifier` (String(100), unique, NOT NULL), `display_name` (String(255), NOT NULL), `is_active` (Boolean, default true), `created_at` (DateTime with TZ). Initial agents seeded in the same Alembic migration via `op.bulk_insert()` after `op.create_table()`.

**Rationale**: Matches the existing model conventions in this codebase exactly â€” `Project` and `Job` both use `UUID(as_uuid=True)` with `func.gen_random_uuid()` as `server_default`, `DateTime(timezone=True)` with both a Python-side `default` and a `server_default=func.now()`, and `Mapped` annotations. Seeding in the migration keeps schema history consistent with data state; admins can add/deactivate agents via SQL without a code deploy.

**Alternatives considered**:
- Enum-based hardcoding: rejected â€” code deploy required for every agent addition/removal.
- Separate seeding script at app startup: rejected â€” couples data initialisation to runtime; ambiguous rollback story.
- `op.execute(sa.text("INSERT INTO agents ..."))`: rejected in favour of `op.bulk_insert()` which is type-safe and easier to review.

## Decision 2: Migration Strategy (two migrations)

**Decision**: Two sequential migrations:
- `0005_add_agents_table.py`: Create `agents` table + seed two initial agents (`spec_driven_development`, `generic_testing`).
- `0006_add_agent_id_to_jobs.py`: Add nullable `agent_id UUID FKâ†’agents.id` to `jobs` with an explicit, named FK constraint.

**Rationale**: Migration 0004 (`0004_add_job_lease_fields.py`) is the direct precedent for adding nullable columns â€” no `server_default`, no backfill, instantaneous PostgreSQL metadata operation. Named FK constraints (`fk_jobs_agent_id`) prevent autogenerated names from drifting between SQLAlchemy versions, making `downgrade()` reliable.

**Alternatives considered**:
- Single migration for both: rejected â€” harder to roll back selectively; violates single-responsibility for migrations.
- Backfill existing `jobs` rows with a default `agent_id`: rejected â€” no stable default UUID exists (agents are DB-generated); NULL is intentional and valid for legacy rows.

## Decision 3: API Field Deprecation Strategy

**Decision**: Mark `dev_agent_type` and `test_agent_type` as `deprecated: true` using `Field(deprecated=True, default=...)` in Pydantic (requires Pydantic â‰Ą 2.7; fallback: `json_schema_extra={"deprecated": True}`). Both fields become optional by adding defaults, so existing callers who omit them still work. New `agent_id: uuid.UUID` field added as required to `CreateTaskRequest`.

**Rationale**: OpenAPI 3.1 Â§4.8.25.3 â€” `deprecated: true` signals transition without breaking existing callers. After `task generate`, the regenerated TypeScript client emits JSDoc `@deprecated` annotations on these fields, giving IDE warnings in the web layer. Removing the fields immediately would break existing callers.

**Fallback**: If Pydantic < 2.7, use `json_schema_extra={"deprecated": True}` alongside the field definition.

**Alternatives considered**:
- Remove fields immediately: breaking change for any caller not yet updated.
- Keep fields required without deprecation: silently confusing when `agent_id` is introduced alongside them.
- `x-deprecated-since` custom extension: useful for docs generators but not universally supported; keep it simple.

## Decision 4: Optional JSON Body on POST /tasks/{id}/push

**Decision**: New `PushTaskRequest` Pydantic model with `git_url: str | None = None`. Route parameter annotated as `Annotated[PushTaskRequest | None, Body()] = None` so the body is entirely optional â€” callers can omit it, send `{}`, or send `{"git_url": "..."}`.

**Rationale**: US3 requires the ability to supply a `git_url` at push time for tasks that had no URL at submission. `Annotated[... | None, Body()] = None` is the FastAPI v0.95+ canonical pattern for optional bodies. When `git_url` is provided, the service saves it to `project.git_url` before executing the push (satisfying "system saves the URL to the project").

**Alternatives considered**:
- Query parameter: semantically incorrect on a mutating `POST`; inconsistent with this API's body-only convention.
- Always require a body: breaks existing callers who currently send no body.

## Decision 5: Project Name in CreateTaskRequest

**Decision**: Add optional `project_name: str | None = None` to `CreateTaskRequest`. A Pydantic `@model_validator(mode="after")` raises a `ValueError` if `project_type == "new"` and `project_name` is missing/empty. FastAPI translates this to a 422 Unprocessable Entity response with a field-level error.

**Rationale**: Cross-field validation belongs in the schema layer. `@model_validator` is the Pydantic v2 mechanism for this, consistent with how `requirements` min_length is handled. No DB migration is needed â€” `Project.name` is already nullable; the constraint is purely at the API boundary.

**Alternatives considered**:
- Two separate request schemas (`CreateNewProjectTaskRequest` / `CreateExistingProjectTaskRequest`): more type-safe but splits a single endpoint into two shapes, adding complexity for one conditional field.
- Required field with no default: breaks existing callers who submit existing-project tasks without sending `project_name`.

## Decision 6: Agent Summary in Task Responses

**Decision**: Add `agent: AgentSummary | None` to `TaskResponse` and `TaskDetailResponse`. `AgentSummary = {id, identifier, display_name}`. `null` for legacy tasks that predate the agent registry (no `agent_id` stored). Keep deprecated `dev_agent_type` / `test_agent_type` in responses as deprecated nullable fields.

**Rationale**: The UI needs the agent's display name to render the task list and detail pages without a second API call. This follows the established `ProjectSummary` inline pattern already used in `TaskResponse`. Null is valid for legacy tasks.

**Alternatives considered**:
- Return only `agent_id` and let clients do a second `GET /agents/{id}` lookup: extra roundtrip; inconsistent with `ProjectSummary` pattern.
- Always return agent display name from the deprecated string field: doesn't work for new tasks that only have `agent_id`.

## Decision 7: OpenAPI Version Bump

**Decision**: Bump `info.version` from `0.2.0` to `0.3.0` (minor version â€” new required field `agent_id` in `CreateTaskRequest` is a consumer-breaking change; `project_name` is additive but conditional-required).

**Rationale**: Constitution Principle VI requires semantic versioning on `info.version` for consumer-facing API changes. Adding a required field (`agent_id`) is a breaking change for existing callers. After `task generate`, the TypeScript client is regenerated from the new spec.

**Alternatives considered**:
- Patch bump (0.2.x): incorrect â€” a new required field is breaking by SemVer definition.
- Major bump (1.0.0): premature; the existing fields are not being removed, only deprecated.
