## PostgreSQL Schema

All tables use UUID primary keys generated by the database. Timestamps are stored with timezone
(`TIMESTAMPTZ`). Status fields use `VARCHAR` with application-level enum validation via SQLAlchemy.

### Table: `projects`

Represents a software initiative the agent works on.

```sql
CREATE TABLE projects (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name        VARCHAR(255) NOT NULL,
    source_type VARCHAR(20)  NOT NULL CHECK (source_type IN ('new', 'existing')),
    git_url     TEXT,
    status      VARCHAR(20)  NOT NULL DEFAULT 'active'
                    CHECK (status IN ('active', 'archived')),
    created_at  TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ  NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_projects_status ON projects(status);
```

**Validation rules**:
- `source_type = 'existing'` MUST have a non-null `git_url`.
- `name` MUST be non-empty.

**State transitions**:
```
active → archived
```

---

### Table: `jobs`

Represents a single requirement submission processed by the agent worker.

```sql
CREATE TABLE jobs (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    project_id      UUID         NOT NULL REFERENCES projects(id),
    requirement     TEXT         NOT NULL,
    status          VARCHAR(20)  NOT NULL DEFAULT 'queued'
                        CHECK (status IN ('queued', 'in_progress', 'completed', 'failed')),
    created_at      TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    started_at      TIMESTAMPTZ,
    completed_at    TIMESTAMPTZ,
    error_message   TEXT
);

CREATE INDEX idx_jobs_project_id  ON jobs(project_id);
CREATE INDEX idx_jobs_status      ON jobs(status);
CREATE INDEX idx_jobs_created_at  ON jobs(created_at DESC);
```

**Validation rules**:
- `requirement` MUST be non-empty (minimum 10 characters).
- At most one job may have `status = 'in_progress'` at any time (enforced by application layer).

**State transitions**:
```
queued → in_progress → completed
                     ↘ failed
```

---

### Table: `work_directories`

Tracks the isolated file system directory allocated for each job's agent execution.

```sql
CREATE TABLE work_directories (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    job_id      UUID NOT NULL UNIQUE REFERENCES jobs(id),
    path        TEXT NOT NULL UNIQUE,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

**Validation rules**:
- `path` MUST be a subdirectory of the configured `AGENT_WORK_PARENT` (enforced by application).
- One-to-one with `jobs` (UNIQUE constraint on `job_id`).

## Entity Relationships

```text
projects ──< jobs >── work_directories
  1          N              1
```

- A **Project** has zero or more **Jobs**.
- Each **Job** has zero or one **WorkDirectory** (created when the job transitions to `in_progress`).

## Redis Key Space

Redis is used for live cross-component state only. Durable records are always in PostgreSQL.

### Job Queue

| Key | Type | Description |
|-----|------|-------------|
| `jobs:queue` | List | FIFO queue of pending job payloads (LPUSH by api, BLMOVE by worker) |
| `jobs:inflight` | List | Jobs currently being processed (BLMOVE destination; crash recovery) |

Queue payload (JSON string in each list element):
```json
{
  "job_id": "<uuid>",
  "project_id": "<uuid>",
  "requirement": "<text>",
  "work_dir": "/agent-work/<job_id>"
}
```

### Job Live State

| Key | Type | TTL | Description |
|-----|------|-----|-------------|
| `job:{job_id}:state` | Hash | 7 days after completion | Mutable job state fields |
| `job:{job_id}:logs` | List | 7 days after completion | Append-only execution log entries |
| `job:{job_id}:updates` | Pub/Sub channel | N/A (not a stored key) | Signal channel for state changes |

**Hash fields for `job:{job_id}:state`**:

| Field | Type | Notes |
|-------|------|-------|
| `status` | string | Mirrors PostgreSQL status |
| `progress` | integer string | 0–100; use `HINCRBY` for atomic increment |
| `worker_id` | string | Hostname or UUID of worker instance |
| `started_at` | integer string | Unix timestamp |
| `completed_at` | integer string | Unix timestamp |
| `error` | string | Error message (set on failure) |

**Log entry format for `job:{job_id}:logs`** (each RPUSH element is a JSON string):
```json
{
  "timestamp": "2026-03-02T13:00:00Z",
  "level": "INFO",
  "message": "Planner produced 3 subtasks"
}
```

## Agent Work Directory Layout

The parent directory is configured via `AGENT_WORK_PARENT` environment variable
(default: `/agent-work` inside the worker container; bind-mounted in dev).

```text
/agent-work/
├── <job_id_1>/       # created when job starts; job_id is the UUID from PostgreSQL
│   ├── workspace/    # cloned repository or new project files
│   ├── context/      # job metadata written by worker (requirement.txt, plan.json)
│   └── output/       # generated artefacts
├── <job_id_2>/
│   └── ...
└── ...
```

Each subdirectory is created atomically when the job transitions to `in_progress`. The path
`/agent-work/<job_id>` is recorded in the `work_directories` table and in the Redis queue payload.

## SQLAlchemy Model Notes

- All models use `mapped_column()` (SQLAlchemy 2.x declarative style).
- Async session via `AsyncSession` + `create_async_engine` with `asyncpg` driver.
- Alembic manages all schema migrations; the initial migration creates all three tables.
- `updated_at` on `projects` is maintained by an `onupdate=func.now()` server default.
